## Documentación
La lógica del validador es simple. Primero se verifica que tanto el input como el output no sean vacíos, si alguno lo es, es una transacción inválida porque no tiene sentido una transacción donde doy y nadie recibe o viceversa. El segundo paso es comenzar a recorrer los diferentes inputs; en esta ocasión se verifica que el utxo usado actual exista y que ya no haya sido usado. Si no lo fue, se verifica que la firma tenga validez. Luego, para cada input se suma el amount del utxo y se guarda para poder comprobar posteriormente que se vuelva a utilizar en la transacción. El tercer paso consiste en recorrer los outputs verificando que el amount no sea igual o menor a cero; si no lo es, se suma al amount de los outputs. Por último, se verifica que la cantidad (amount) de los inputs y outputs coincida.

Por el lado de las dificultades enfrentadas al momento de realizar el código, estuvieron presentes aquellos requisitos que se debían cumplir para que una transacción fuera válida. En ese caso fueron de gran ayuda los test, ya que daban una orientación de dichos requisitos. Como que la transacción no tuviera outputs negativos o que la suma de las cantidades de entrada y salida fueran las mismas. También se debió prestar atención a los tipos de datos, pero leyendo el código fue fácil saber a qué atributo se debía buscar.
